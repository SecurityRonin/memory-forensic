Recover the kernel DTB (page-table root) from a header-less raw dump.

For dumps with no crash-dump metadata (no embedded CR3, no PsLoadedModuleList), scan_for_kernel has nothing to anchor the DTB descent on. This entry point recovers the kernel DTB directly from raw physical memory:

  1. Enumerate self-referencing PML4 candidates (enumerate_self_ref_pml4s). On a real dump this surfaces the kernel DTB and many process DTBs — all self-reference at the same canonical index (220 on SecurityNik), so the set is ambiguous.
  2. Order candidates by ascending physical address and accept the first whose page tables map an ntkrnlmp/ntoskrnl PE with a valid RSDS GUID (locate_kernel_via_dtb_only). Verification rejects self-referencing pages that are not page-table roots; the lowest-physical ordering selects the kernel DTB among the process DTBs (whose shared kernel half also maps the kernel, so verification alone would not distinguish them).

Returns the physical address of the kernel DTB, or None if no candidate verifies. Panic-free and bounded against crafted dumps.
