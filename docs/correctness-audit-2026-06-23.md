# memf correctness audit — findings (2026-06-23)

Multi-agent audit of memory-forensic against Volatility 3, Rekall, and MemProcFS. Each finding adversarially verified against the reference source before inclusion.

**Result:** 37 confirmed of 40 raw — {'critical': 2, 'high': 18, 'medium': 10, 'low': 7}

**COVERAGE CAVEAT — PARTIAL AUDIT.** Of 16 subsystems, several did NOT complete: `secrets-keys` and `injection-hooks` were blocked by usage-policy cyber safeguards; `core-translation` (the page-table/DTB foundation), `format-readers`, `pool-object-infra`, `registry-core`, `services-drivers`, `sessions-ui`, `vad-modules` were rate-limited and failed. The 37 findings below are from the ~7 subsystems that completed; a full re-run (especially core-translation) would likely surface more.

## Recurring root causes (one fix → many findings)

1. **Wrong `_CM_KEY_NODE`/`_CM_KEY_VALUE` offsets (shifted +4/+2)** in lsadump.rs, cachedump.rs, sam.rs — while the repo's *validated* hashdump.rs uses the correct ones (0x14/0x1c/0x48/0x4c, value 0x02/0x04/0x08/0x14). Copy-pasted wrong layout. A single shared registry-cell reader fixes all.
2. **Missing `ri` (index-root) subkey-list type** in the shared registry subkey walker — breaks navigation of large keys (CLSID, InventoryApplication, Run). Hits amcache, com-hijacking, run-keys, lsadump, cred-lsa. One fix.
3. **Missing/absent credential decryption** — cachedump (NL$/DCC2), lsadump (LSA secrets), wdigest (LsaProtectMemory) emit ENCRYPTED bytes as if plaintext = fabrication.
4. **Missing object types/tags** — network (TcpL/UdpA/IPv6), Linux (LISTEN/UDP), psxview (Win10 `Pro\xe3`), registry (`ri`).
5. **Over-strict gates / silent-empty** — walk_processes aborts the whole list on one unreadable field (the owner-gate class).

### [HIGH] cred-hashes — hashdump rev-2 (AES) NT hash uses LM salt/ciphertext offsets (+4/+20) instead of NT offsets (+8/+24) — wrong NT hash on modern Windows
- **class:** wrong-or-missing-offset
- **memf:** crates/memf-windows/src/hashdump.rs:539-547 (decrypt_user_hash rev-2 branch: salt = blob[offset+4..offset+20], enc = blob[offset+20..offset+52]) applied identically to NT via extract_hashes_from_v:498-506
- **reference:** Volatility3 hashdump.py get_user_hashes: NT rev-2 uses nt_salt = sam_data[nt_offset+8 : nt_offset+24] and enc_nt_hash = sam_data[nt_offset+24 : nt_offset+56] (LM rev-2 uses +4/+20). impacket SAM_HASH_AES is PekID(2)+Revision(2)+DataOffset(4)+Salt(16)+Hash, i.e. salt at +8 and ciphertext at +24 — matching vol3's NT path. memf uses the +4/+20 window for the NT hash, so it AES-decrypts the wrong 32-byte ciphertext with the wrong 16-byte salt/IV.
- **impact:** On Win10 1607+ dumps (rev-3 AES hbootkey + rev-2 per-hash, the common modern case), the recovered NT hash is corrupt — it will not match the real account hash, defeating pass-the-hash target identification and password-reuse correlation. The empty-hash sentinel masks it as a plausible-looking but incorrect 32-hex value. The bug is untested: only a rev-1 (RC4) golden vector exists; there is no rev-2 NT golden vector.

### [HIGH] cred-lsa — lsadump emits RAW ENCRYPTED LSA-secret bytes as if they were the secret value (no LSA-key / AES / DES / RC4 decryption)
- **class:** fabrication-or-silent-empty
- **memf:** crates/memf-windows/src/lsadump.rs:351-417 (read_currval_data) and :234-242 (walk_lsa_secrets fills LsaSecretInfo.data with the raw CurrVal bytes)

### [HIGH] cred-lsa — WDigest password is reported as cleartext/encrypted-passthrough but is LsaProtectMemory-encrypted on ALL modern Windows, not only 8.1+ — reference tools decrypt it; memf never does
- **class:** fabrication-or-silent-empty
- **memf:** wdigest.rs:69-82 (reads password UNICODE_STRING raw, then decode_utf16le_or_none: if it decodes as UTF-16 → returns it as plaintext `password`, else stuffs raw bytes into `password_encrypted`); module doc :6-13 claims cleartext on Win7/8 and only-encrypted on 8.1+.

Analyst either trusts a bogus password or wrongly concludes 'no wdigest creds'.

### [HIGH] kernel-base — KdDebuggerDataBlock used without OwnerTag ('KDBG') validation — accepts an unvalidated/encoded block
- **class:** physical-vs-virtual-or-weak-validation
- **memf:** ~/src/memory-forensic/crates/memf-windows/src/sysinfo.rs:75-88 (matches on symbol_address("KdDebuggerDataBlock") then reads fields directly, no Header.OwnerTag check)
- **reference:** Volatility3 info.py:227 gates all KDBG-derived output on `if kdbg.Header.OwnerTag == 0x4742444B` ('KDBG'); Rekall kdbgscan.py:43-59 scans by the OwnerTag and validates the block before trusting its fields. memf reads KDBG fields with no signature/OwnerTag check. ### [HIGH] linux — TCP connection walker misses all LISTEN sockets and all UDP endpoints: walks only `tcp_hashinfo.ehash`
- **class:** missing-object-type-or-tag
- **memf:** crates/memf-linux/src/network.rs:12-65 (walk_connections, iterates only ehash buckets) and 105-154 (walk_connections6, same)
- **reference:** In Linux, `inet_hashinfo.ehash` holds sockets only in the ESTABLISHED..CLOSING states; LISTEN-state sockets live in a separate hash (`inet_hashinfo.lhash2`, or pre-4.x `listening_hash`), and UDP sockets are in `udp_table.hash/hash2` — none of which memf reads. Volatility3 ~/src/_refs/volatility3/volatility3/framework/plugins/linux/sockstat.py enumerates sockets by walking each task's open file descriptors (`list_sockets`, lines ~540-570) and dispatching by family (`SockHandlers._inet_sock` handles AF_INET/AF_INET6 in ALL states including LISTEN, and `child_sock.get_protocol()` yields TCP/UDP),
- **impact:** An analyst listing network connections sees only established/transitional TCP and never sees a malware C2 listener (a bind-shell or backdoor in LISTEN state) or any UDP beacon — exactly the artifacts most relevant to intrusion triage. The output also lacks the owning PID that the reference associates, breaking process-to-socket attribution.

### [HIGH] network — Windows network walker misses TcpL/_TCP_LISTENER and UdpA/_UDP_ENDPOINT (and TTcb) — only _TCP_ENDPOINT is enumerated
- **class:** missing-object-type-or-tag
- **memf:** crates/memf-windows/src/network.rs:199 (TCPE_TAG = b"TcpE" is the only pool tag) and :449-618 scan_tcp_endpoints / :30-77 walk_tcp_endpoints; repo-wide there is no UDP or TCP-listener enumeration (grep for UdpA/_UDP_ENDPOINT/TcpL/_TCP_LISTENER in crates/memf-windows finds nothing but pool_tag descriptions).
- **reference:** Volatility3 plugins/windows/netscan.py:84-117 create_netscan_constraints builds THREE pool constraints — b"TcpL"→_TCP_LISTENER, b"TcpE"→_TCP_ENDPOINT, b"UdpA"→_UDP_ENDPOINT — plus b"TTcb"→_TCP_ENDPOINT for build 20348 (lines 108-117). netscan.py:406-479 then renders UDP endpoints (UdpA, dual_stack_sockets, state blank), TCP endpoints, AND TCP listeners (TcpL, state 'LISTENING', remote port 0). - **class:** over-strict-gate
- **memf:** crates/memf-windows/src/network.rs:535 `if af as u16 != AF_INET { continue; }` (scan path); :485-490 read_ipv4 reads only 4 bytes; protocol hardcoded "TCPv4" at :600 and :103. walk_tcp_endpoints path (:137-156 addresses_from_addr_info) reads _ADDR_INFO.Remote as u32 and _LOCAL_ADDRESS.pData as 4 bytes only, with protocol "TCPv4" at :103.
